项目目标

从动漫视频中自动截取所有符合条件的 15 秒片段。

合格片段的判定标准：
1. 每个片段长度固定为 15 秒。
2. 15 秒内需要出现大于等于 13 次动漫人脸出现事件。
3. 人脸表情需要尽量清晰。
4. 允许同一帧画面中同时计入多个人脸。
5. 允许同一角色重复计数。
6. 当前只考虑所有动漫视频，不考虑真人视频。

输入视频：
data/1.mp4

运行环境：
1. Windows 10/11。
2. 使用 Python 虚拟环境。
3. Python 可执行文件路径：
   D:\Programs\DevEnvironments\Anaconda\anaconda3\envs\myenv\python.exe
4. 有 GPU 硬件。
5. 涉及 PowerShell 读取文件时使用 UTF-8，避免中文乱码。


核心定义

统计对象不是“不同角色”，而是“人脸出现事件”。

人脸出现事件的定义：
1. 使用轻量 IoU 跟踪器，把相邻帧中位置重叠度较高的人脸框连接成同一条轨迹。
2. 一条轨迹代表一次人脸出现事件。
3. 同一角色离开画面后再次出现，如果中间轨迹断开，则计为新的出现事件。
4. 这符合当前“同一角色可以重复计数”的需求。

后续如果要改成“15 秒内必须出现 13 个不同角色”，需要在现有轨迹对象基础上增加角色识别或特征聚类能力。


整体处理流程

1. 使用 ffmpeg 对视频抽帧。
   建议抽帧间隔为 0.3 秒，约等于每秒 3.33 帧。

2. 对每一帧运行动漫人脸检测器。
   输出每张脸的：
   - bbox，人脸在画面中的位置框
   - confidence，检测置信度
   - label，检测类别标签，例如 anime_face

3. 对检测结果做质量过滤。
   过滤依据包括：
   - 人脸框大小是否足够
   - 人脸区域是否清晰
   - 检测置信度是否足够高

4. 使用 IoU 跟踪器连接相邻帧中的同一张脸。
   得到结构化的人脸轨迹列表。

5. 根据轨迹起始时间统计人脸出现事件。
   用 15 秒滑动窗口扫描时间轴：
   - 如果窗口内轨迹起始数大于等于 13，则该窗口合格。

6. 输出所有符合条件且互不重叠的 15 秒片段。
   建议使用贪心策略：
   - 找到一个合格窗口后，以该窗口起点截取 15 秒。
   - 然后跳到该片段结束时间之后继续搜索。

7. 使用 ffmpeg 按时间段截取视频片段。


动漫人脸检测器

优先使用专门的动漫人脸检测器。

推荐方案：
使用 dghs-imgutils 的动漫脸检测能力。

原因：
1. 底层基于 YOLOv8。
2. Windows 上可以通过 pip 安装。
3. 不需要手动查找和管理 yolov8-animeface 权重。
4. 可以自动使用 GPU。
5. 对动漫脸的准确率和召回率通常比通用人脸检测器更好。

备选方案：
使用基于 YOLO 的动漫脸检测权重，例如 yolov8-animeface。


表情清晰度过滤

为了让截取片段中的人脸表情尽量清晰，需要过滤低质量人脸。

建议过滤条件：
1. 人脸框足够大。
   例如人脸高度至少占画面高度的 4.5%。
   这样可以排除远景中很小、无法看清表情的人脸。

2. 人脸区域足够清晰。
   对人脸裁剪区域计算拉普拉斯方差，用于过滤模糊或动态糊的人脸。

3. 检测置信度足够高。
   侧脸、遮挡严重、画面质量差的人脸通常置信度较低。

说明：
清晰度阈值需要结合具体视频实测后微调。
第一版可以先使用保守经验值，跑一遍后再根据结果调整。


默认参数

抽帧间隔：
0.3 秒
含义：太密会浪费算力，太疏可能漏掉快速切镜中的人脸。

置信度下限：
0.5
含义：太高会漏检，太低会混入误检。

最小脸高：
占画面高度 4.5%
含义：远景小脸不计入“表情清晰”的人脸。

清晰度下限：
经验值，需实测微调
含义：过滤模糊脸或动态糊脸。

IoU 阈值：
0.3
含义：判断相邻帧中的两个人脸框是否属于同一条轨迹。

窗口长度：
15 秒
含义：每个候选片段的固定长度。

窗口内最少出现事件数：
13
含义：15 秒内至少出现 13 次人脸出现事件。


架构扩展要求

检测阶段需要设计成可扩展结构：
1. 可以注册多个检测器。
2. 每个检测器返回统一格式的检测框。
3. 每个检测框都带类别标签 label。
4. 后续新增检测目标时，只需要新增检测器和标签，不需要改动后面的跟踪、计数和截取逻辑。

示例：
1. 当前检测器：
   label = anime_face

2. 未来可扩展：
   label = animal
   label = object
   label = other_face_type


轨迹对象要求

跟踪阶段不能只返回“出现了多少次”，需要返回结构化轨迹对象。

每条轨迹至少包含：
1. track_id
2. label
3. start_time，轨迹开始时间
4. end_time，轨迹结束时间
5. detections，轨迹包含的检测框列表
6. representative_frame，代表帧信息
7. representative_crop，代表帧中的脸部裁剪图路径，或者保存其在视频中的位置，供后续阶段回到视频中取图提特征

这样后续可以支持：
1. 不同角色识别。
2. 人脸特征提取。
3. 去重计数。
4. 人脸质量评分。
5. 对不同画风选择不同模型。


批量处理要求

代码需要支持批量识别多个视频。

要求：
1. 模型只加载一次。
2. 多个视频依次复用同一个检测器实例。
3. 每个视频独立输出检测结果、轨迹结果和截取片段。
4. 所有可调参数集中放在配置区或配置文件中。


不同画风适配

不同动漫画风可能需要不同参数或模型。

第一版可以通过配置手动指定参数。

后续可以扩展为：
1. 从视频中间抽取若干帧。
2. 对画风或检测效果进行投票。
3. 自动选择更合适的模型或参数组。


第一版建议范围

第一版先实现：
1. 单视频输入。
2. ffmpeg 抽帧。
3. dghs-imgutils 动漫人脸检测。
4. 人脸大小、置信度、清晰度过滤。
5. IoU 轻量跟踪。
6. 统计 15 秒窗口内的人脸出现事件。
7. 输出互不重叠的合格片段。
8. 使用 ffmpeg 截取片段。
9. 保留批量处理和多检测器扩展接口。

后续再实现：
1. 批量视频处理。
2. 不同角色去重。
3. 多模型或多画风自动选择。
4. 更复杂的质量评分。
